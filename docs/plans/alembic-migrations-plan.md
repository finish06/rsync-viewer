# Implementation Plan: Alembic Database Migrations

**Spec Version**: 0.1.0
**Created**: 2026-03-01
**Team Size**: Solo
**Estimated Duration**: 2–3 hours

## Overview

Replace `SQLModel.metadata.create_all()` with Alembic for versioned database schema management. Migrations run automatically on app startup and during Docker deployments. A baseline migration captures the full current schema. Tests continue using `create_all()` unchanged.

## Objectives

- Initialize Alembic with SQLModel metadata integration
- Create a baseline migration representing the current schema
- Replace `create_all()` in app startup with `alembic upgrade head`
- Update Docker entrypoint to run migrations before uvicorn
- Maintain existing test behavior (no Alembic dependency in tests)

- [ ] All 12 acceptance criteria implemented
- [ ] Fresh database fully created by `alembic upgrade head`
- [ ] Existing production database stampable without alteration
- [ ] `alembic revision --autogenerate` correctly detects model changes
- [ ] All existing tests pass without modification
- [ ] Quality gates passing (ruff, pytest)

## Architecture Notes

**Database URL**: Read from `app.config.Settings.database_url` — Alembic's `env.py` imports this directly rather than parsing `alembic.ini`'s `sqlalchemy.url`.

**SQLModel metadata**: All models register on `SQLModel.metadata`. The `env.py` sets `target_metadata = SQLModel.metadata` after importing all model modules.

**Model imports**: Currently `app/models/__init__.py` only exports `SyncLog` and `ApiKey`. The `env.py` must import all model modules to ensure their tables are registered on the metadata before autogenerate runs. Models: `sync_log`, `monitor`, `failure_event`, `webhook`, `webhook_options`, `notification_log`, `smtp_config`, `oidc_config`, `user`.

**Test isolation**: Tests use `SQLModel.metadata.create_all(engine)` in `conftest.py:54` with a PostgreSQL test database and per-test transaction rollback. This remains unchanged — Alembic is not involved in the test lifecycle.

**Docker entrypoint**: Currently at `entrypoint.sh` (project root), it auto-generates `ENCRYPTION_KEY` then `exec "$@"`. Migration command will be added before the exec.

## Production Migration Note

For the existing production database (which already has all tables from `create_all()`):

```bash
# One-time command — stamps the DB as "at baseline" without running the migration
alembic stamp head
```

This creates the `alembic_version` table with the baseline revision ID. No ALTER TABLE commands are executed. Document this in the PR description.

## Dependency Note: User Preferences

The `specs/user-preferences.md` spec (AC-012) requires an Alembic migration to add the `preferences` column. **This feature should be implemented first**, so the user-preferences feature can use `alembic revision --autogenerate` to create its migration.
